# seed.json 修正ガイド

`src/data/seed.json` は RAG 診断の知識ベースとなるデータです。症状テキストから部品を検索する精度（Recall@3 / Accuracy@1）に直結するため、追加・修正・削除の際は以下の指針に従ってください。

## 1. ファイルの役割

- `POST /diagnose` で受け取った症状をベクトル化し、Vectorize から類似度 Top-3 を検索します。
- 検索結果の ID で D1 `diagnoses` テーブルからメタデータを取得し、LLM へ渡します。
- `seed.json` は D1 と Vectorize の両方の投入元となり、`npm run seed` で再同期されます。

## 2. レコード形式

```json
{
  "id": "battery_001",
  "symptom": "秒針が5秒おきに飛んで止まる",
  "part": "電池",
  "description": "電池電圧が低下すると、秒針が5秒間隔で飛んで動くICの省電力モードが発動します。"
}
```

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | string | 重複しないユニーク ID。`{部品英名}_{連番}` を推奨。 |
| `symptom` | string | ユーザーが入力しそうな症状表現。短く具体的に。 |
| `part` | string | 推定する故障部品名。RAG の正解ラベルとしても使われる。 |
| `description` | string | 症状・部品・原因の解説。LLM への文脈情報。 |

## 3. ID の命名規則

- 形式: `{カテゴリ}_{3桁の連番}`
- カテゴリは部品の英名を小文字で統一します（例: `battery`, `gear`, `calendar`, `crown`）。
- 新規追加時は既存 ID と重複しないよう、同カテゴリ内の最大連番 +1 を使ってください。
- ID は投入後に変更すると Vectorize 内の古いベクトルと不整合が起きるため、`npm run seed` を実行し直してください。

## 4. symptom の書き方

- **ユーザー視点の言葉**を使う。専門用語だけでなく「止まった」「遅れる」「音がする」などの表現を含める。
- 1 レコードに対しては **1 つの症状**を書く。複数症状を羅列しない。
- 原因や部品名は `description` に任せ、`symptom` には観察された事象だけを書く。
- 同じ部品でも複数の言い回しをカバーするために、症状の揺らぎごとにレコードを分けて追加すると検索精度が上がります。

## 5. description の書き方

- 症状がなぜその部品に結びつくのか、簡潔に説明する。
- 類似症状で別部品の可能性もある場合は、「〜の場合もあるため、他の症状と合わせて判断します」のように注記する。
- 機能故障と外観不良を区別する場合は明記する（例: 「機能故障ではなく外観不良です」）。

## 6. part の選び方

- `part` は診断結果として返す部品名です。可能な限り既存の部品名を再利用し、ばらつきを避けてください。
- 新しい部品カテゴリを追加する場合は、`id` のカテゴリ名もそれに合わせます。

## 7. 修正後の反映手順

1. `src/data/seed.json` を編集する。
2. 必要に応じて `tests/eval.json` の評価ケースも追加・修正する。
3. ローカルまたはリモートで反映する:

```bash
# .env / .dev.vars の GROQ_API_KEY などが整っている状態で
npm run seed
```

> `npm run seed` は **Vectorize インデックスと D1 テーブルをクリアして再投入**します。PoC 段階では本番インデックスもそのまま上書きされるため注意してください。

4. 反映後に評価を実行し、品質が低下していないか確認する:

```bash
# 別ターミナルで wrangler dev --remote を起動した状態で
npm run eval
```

## 8. 品質を高めるコツ

- **Recall@3 を上げる**: 同じ部品に対し、ユーザーの言い回しのバリエーションを増やす。
- **Accuracy@1 を上げる**: 部品間で症状が紛らわしいケースには `description` で区別理由を明記し、判定材料を増やす。
- **迷ったら追加**: 症状が曖昧でも、部品が明確ならレコードを追加。自信がない場合は `part` を広めのカテゴリ（例: `ムーブメント`）にして、誤訳を減らす。

## 9. 注意事項

- `id` は後から変更しない。変更が必要な場合は `npm run seed` で再同期する。
- JSON の文法エラーがあると `npm run seed` が失敗する。編集後は `node --check src/data/seed.json` などで構文を確認するとよい。
- 機密情報や個人情報を `seed.json` に含めない。
