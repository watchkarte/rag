# WatchKarte 人間向け解説書

このドキュメントは、技術的な仕様書（`spec.md` / `spec_mvp.md` / `architecture.md`）を読まなくても、**WatchKarte がどうやって動いていて、お金はどれくらいかかるのか**を把握できるように書いたものです。

---

## 1. WatchKarte とは何か

**WatchKarte**（ウォッチカルテ）は、クォーツアナログ時計の故障症状を入力すると、最も可能性の高い部品と「次にすべきこと」を提案する Web サービスです。

例えば、

> 「秒針が 5 秒おきに飛んで止まる」

と入力すると、

> 「電池」の可能性が高い → 「電池を新品に交換してください」

という診断結果が返ってきます。

技術的には「RAG（Retrieval-Augmented Generation）」という仕組みを使っています。簡単に言うと、**「あらかじめ用意した時計修理の知識データから似た症状を探し、LLM に判断を任せる」**方式です。

---

## 2. 診断までの流れ（ざっくり）

ユーザーが症状を入力してから結果が返るまで、おおまかに以下の流れになります。

```
┌─────────────────────────────────────────────────────────────────┐
│  1. ユーザーが UI または API で症状を入力する                     │
│     例: 「秒針が5秒おきに飛んで止まる」                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  2. 症状テキストをベクトル（数値の列）に変換する                  │
│     サービス: Cloudflare Workers AI（bge-m3）                    │
│     これによって「似た意味の文章」を機械が検索できるようになる    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  3. ベクトルデータベースで似た症状を探す                          │
│     サービス: Cloudflare Vectorize                               │
│     現在は 60 件の診断データの中から Top-3 を探す                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  4. 似た症状の詳細情報を通常のデータベースから取得する            │
│     サービス: Cloudflare D1（SQLite）                            │
│     症状の説明や部品名、修理の難易度などを取得                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  5. 検索結果 + ユーザー症状を LLM に渡す                          │
│     サービス: Groq API（llama-3.1-8b-instant）                   │
│     AI が「どの部品が怪しいか」「次に何をすべきか」を日本語で生成 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  6. 診断結果を JSON または UI 画面で返す                          │
│     { part, confidence, reason, nextAction }                     │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 使われているサービスと役割

| サービス | 役割 | このアプリでの用途 |
|---|---|---|
| **Cloudflare Workers** | サーバーレスアプリの実行環境 | Hono で書いた API/UI を動かす |
| **Cloudflare D1** | SQLite データベース | 症状・部品・説明などのメタデータを保存 |
| **Cloudflare Vectorize** | ベクトルデータベース | 症状のベクトル同士の類似度検索 |
| **Cloudflare Workers AI** | AI 推論サービス | 症状テキストを 1024 次元のベクトルに変換 |
| **Groq API** | 高速 LLM API | 検索結果から最終的な診断文を生成 |

Web ページの UI は `GET /`、診断 API は `POST /diagnose` で提供されています。

---

## 4. 「確信度（confidence）」って何？

レスポンスに含まれる `confidence`（0.0 〜 1.0）は、**LLM が「どのくらい自信があるか」ではありません**。

Vectorize で検索した結果のうち、**最も似ていた症状のスコア**を 0 〜 1 に正規化した数値です。つまり「入力した症状と、事前データの中のどれがどれくらい似ているか」を表しています。

- `confidence >= 0.6` の場合: LLM を呼び出して診断理由や次のアクションを生成
- `confidence < 0.6` または検索結果が 0 件の場合: 「データに無いため診断できませんでした」と返す（LLM は呼ばない）

この 0.6 という閾値は `src/types.ts` の `CONFIDENCE_THRESHOLD` で定義されています。

---

## 5. データはどうやって登録されている？

診断に使う 60 件の初期データは `src/data/seed.json` に入っています。

`npm run seed` を実行すると、以下が行われます。

1. Vectorize の既存ベクトルをすべて削除
2. D1 の `diagnoses` テーブルをクリア
3. `seed.json` の 60 件を D1 に挿入
4. 各データから「症状 + 部品 + 説明」のテキストを作り、Workers AI でベクトル化
5. ベクトルを Vectorize に登録

> ⚠️ 現状の seed は「本番のインデックスを一旦空にして再投入」するため、seed 実行中は診断サービスが一時的に使えなくなります。PoC/MVP 段階ではこの仕様を許容しています。

## 6. ランニングコストの整理

WatchKarte を動かすコストは、主に以下の 5 つから成ります。

1. Cloudflare Workers（ホスティング・リクエスト処理）
2. Cloudflare D1（メタデータ読み書き）
3. Cloudflare Vectorize（ベクトル検索）
4. Cloudflare Workers AI（埋め込み生成）
5. Groq API（LLM 生成）

以下、それぞれの料金体系と、いくつかの利用シナリオでの目安を示します。

### 6.1 各サービスの料金体系（2026 年 7 月時点）

> 出典: [Cloudflare Workers Pricing](https://developers.cloudflare.com/workers/platform/pricing/)、[Cloudflare D1 Pricing](https://developers.cloudflare.com/d1/platform/pricing/)、[Cloudflare Vectorize Pricing](https://developers.cloudflare.com/vectorize/platform/pricing/)、[Cloudflare Vectorize Intro](https://developers.cloudflare.com/vectorize/get-started/intro/)、[Cloudflare Workers AI Pricing](https://developers.cloudflare.com/workers-ai/platform/pricing/)、[Groq Pricing](https://groq.com/pricing/)

#### Cloudflare Workers

| プラン | リクエスト | CPU 時間 |
|---|---|---|
| **Free** | 100,000 / 日 | 1 リクエストあたり 10 ms |
| **Paid**（$5/月） | 1,000 万 / 月を含む、超過分 $0.30 / 100 万 | 3,000 万 ms / 月を含む、超過分 $0.02 / 100 万 ms |

WatchKarte では 1 診断あたり 1 回の Worker リクエストが発生します（静的ページアクセスも 1 リクエスト）。

#### Cloudflare D1

| 項目 | Free | Paid（$5/月） |
|---|---|---|
| Rows read | 500 万 / 日 | 250 億 / 月を含む、超過分 $0.001 / 100 万 |
| Rows written | 10 万 / 日 | 5,000 万 / 月を含む、超過分 $1.00 / 100 万 |
| Storage | 5 GB まで | 5 GB を含む、超過分 $0.75 / GB・月 |

1 回の診断で D1 には `SELECT ... WHERE id IN (...)`（最大 3 件）を 1 回発行するので、**3 rows read / 診断**です。60 件のデータなのでテーブルも小さく、コストは事実上無視できます。

#### Cloudflare Vectorize

| 項目 | Free | Paid（$5/月） |
|---|---|---|
| Queried dimensions | 3,000 万 / 月 | 5,000 万 / 月を含む、超過分 $0.01 / 100 万 |
| Stored dimensions | 500 万 / 月 | 1,000 万 / 月を含む、超過分 $0.05 / 1 億 |

Vectorize は **1024 次元**のベクトルを使います。現在のデータは 60 件なので、

- 保存ベクトル次元数: 60 × 1024 = **61,440 次元**
- 1 回の検索で比較される次元数: 61,440 次元程度（データ件数が少ないためごく微小）

データ件数が少ないうちは、Vectorize のコストもほぼ無視できます。

#### Cloudflare Workers AI（bge-m3）

| 項目 | 料金 |
|---|---|
| 埋め込みモデル `@cf/baai/bge-m3` | **$0.012 / 100 万 input tokens** |
| Free tier | 10,000 neurons / 日 |
| bge-m3 の消費 | 100 万 tokens あたり 1,075 neurons |

1 診断あたり、症状テキストを埋め込みに変換するだけなので、トークン数は数十字〜 100 トークン程度です。10 万診断/月でも数百円以下のレベルです。

#### Groq API（llama-3.1-8b-instant）

| 項目 | 料金 |
|---|---|
| Input tokens | **$0.05 / 100 万 tokens** |
| Output tokens | **$0.08 / 100 万 tokens** |
| Free tier | 14,400 リクエスト/日、30 RPM、30K TPM |

1 診断あたり、プロンプトに 3 件の検索結果 + ユーザー症状を含めるため、入力トークンは約 500〜800 トークン、出力 JSON は約 100 トークン程度です。

### 6.2 利用シナリオ別の目安（月額 USD）

以下はあくまで概算です。実際のトークン数や CPU 時間によって変動します。

| シナリオ | 診断回数/月 | Workers | D1 | Vectorize | Workers AI | Groq | 合計（概算） |
|---|---|---:|---:|---:|---:|---:|---:|
| **Free tier 運用** | 数百〜数千回 | $0 | $0 | $0 | $0 | $0 | **$0** |
| **個人利用・小規模** | 1,000 回 | $5（Paid 最低料金） | ~$0 | ~$0 | ~$0.001 | ~$0.05 | **~$5.05** |
| **中小規模** | 10,000 回 | $5 | ~$0 | ~$0 | ~$0.01 | ~$0.50 | **~$5.51** |
| **中規模** | 100,000 回 | $5 | ~$0 | ~$0.50（超過） | ~$0.12 | ~$5 | **~$10.62** |
| **大規模** | 1,000,000 回 | $5 | ~$0 | ~$10（超過） | ~$1.20 | ~$50 | **~$66** |

※ Workers Paid は 1,000 万リクエスト/月まで含まれるため、100 万診断/月まではリクエスト超過料金は発生しません。100 万を超えると $0.30 / 100 万リクエストが加算されます。

#### 補足: なぜ最低 $5 かかる？

Cloudflare Workers の **Paid プランは月 $5 から**です。Vectorize 自体は Workers Free プランでも利用できますが、実運用では D1 の日次リミットや安定した運用を考慮して **Workers Paid に加入するのが現実的**です。Free プランで運用する場合、D1（rows read 500 万/日、rows written 10 万/日）や Workers（10 万リクエスト/日）の制限に注意が必要です。

### 6.3 コストを抑えるポイント

1. **診断回数を減らす**
   - フロントエンドで入力デバウンスを入れる
   - キャッシュ（Cloudflare AI Gateway 等）を検討する
2. **LLM 呼び出しを減らす**
   - `confidence < 0.6` の場合はすでに LLM を呼ばない設計になっている
   - キャッシュで同じ症状のリクエストを短絡できる
3. **データ件数を増やしても Vectorize コストは線形に増える**
   - 1,000 件 × 1024 次元 = 100 万次元の保存
   - 1 回の検索あたり 100 万次元の比較になるため、検索コストは 60 件の約 17 倍に
4. **Groq の Free tier を活用する**
   - 14,400 リクエスト/日（約 43 万/月）までは無料
   - ただし 30 RPM、30K TPM の制限があるため、混雑時は 429 エラーが出る可能性あり

### 6.4 なぜクレジットカード未登録でも動いているのか？

Cloudflare アカウントにクレジットカードを登録していなくても、本番公開後 3 日経っても Billing に料金が表示されないのは、**すべてのサービスが無料枠内で収まっているから**です。

| サービス | 無料枠 | WatchKarte での消費量（例） |
|---|---|---|
| **Cloudflare Workers** | 100,000 リクエスト/日 | 1 診断 = 1 リクエスト。数千〜数万回/日なら余裕 |
| **Cloudflare D1** | rows read 500 万/日、rows written 10 万/日 | 1 診断 = 3 rows read。60 件のテーブルなら無視できる規模 |
| **Cloudflare Vectorize** | queried 3,000 万 dimensions/月、stored 500 万 dimensions/月 | 60 件 × 1024 次元 = 61,440 stored。1 回の検索 ≒ 62,000 dimensions |
| **Cloudflare Workers AI** | 10,000 neurons/日 | bge-m3 は 100 万 tokens で 1,075 neurons。1 診断で数十 tokens 程度 |
| **Groq API** | 14,400 リクエスト/日、30 RPM、30K TPM | 1 診断 = 1 リクエスト。数千〜数万回/日なら余裕 |

つまり、

- Cloudflare 側は **Workers Free プラン**で運用しており、各サービスの無料枠に収まっている
- Groq 側は **Free tier の API キー**を使っており、無料枠に収まっている
- 両方とも無料枠を超えなければ、クレジットカード未登録でも請求は発生しない

#### 月あたりの無料診断回数目安

WatchKarte の現在のデータ件数（60 件）であれば、各サービスの無料枠を「診断回数」に換算するとおおまかに以下の通りです。

| サービス | 無料枠（月換算） | 備考 |
|---|---|---|
| **Cloudflare Workers** | 約 300 万回/月 | 100,000 リクエスト/日 × 30 日 |
| **Cloudflare D1** | 約 5,000 万回/月 | 5M rows read/日 × 30 日 ÷ 3 rows/診断 |
| **Cloudflare Vectorize** | 約 2.9 万回/月 | 計算式: (Q + 60) × 1024 ≤ 3,000 万 |
| **Cloudflare Workers AI** | 約 555 万回/月 | 10,000 neurons/日 ÷ 0.054 neurons/診断 × 30 日 |
| **Groq API** | 約 43 万回/月 | 14,400 リクエスト/日 × 30 日 |

このうち、もっとも先に無料枠に達するのは **Vectorize の約 2.9 万回/月**です。

#### 注意点

- Workers Free プランは**日次リミット**（24 時間ごとにリセット）なので、1 日で 10 万リクエストを超えるとその日はエラーになる
- Groq も Free tier は**日次リミット**なので、14,400 リクエスト/日を超えると 429 エラーになる
- 無料枠を超えた運用を続けたい場合は、Cloudflare Workers Paid プランへの加入と、Groq の有料プラン（クレジットカード登録）が必要になる

現状、Billing ページに $0 と表示されているのは正常な状態です。請求が発生するのは、Workers Paid プランに加入するか、Cloudflare/Groq の有料枠を超えたときだけです。

## 7. 技術的な注意点（動作を理解する上で重要）

### 7.1 ローカル開発は `--remote` が必要

Vectorize は `wrangler dev` の local モードに対応していません。そのため、

```bash
npm run dev   # 中身: wrangler dev --remote
```

で、Cloudflare 上のリモートリソースを使って開発します。

### 7.2 本番デプロイ後も Groq API キーが必要

コードには `GROQ_API_KEY` が環境変数として必要です。デプロイ後は、

```bash
npx wrangler secret put GROQ_API_KEY
```

で Cloudflare Workers の Secret として登録する必要があります。

### 7.3 監視は限定的

MVP では `wrangler tail` と `console.error` によるログ出力のみです。本格的に運用する場合は、Cloudflare Workers Analytics や外部監視サービスの導入を検討してください。

### 7.4 RAG の精度が全体の鍵

このサービスの品質は、**「入力された症状に対して、正しい部品が Vectorize の Top-3 に入るか」**に大きく依存します。`tests/eval.json` を使った評価で `Recall@3 >= 70%`・`Accuracy@1 >= 60%` を目安にしています。

---

## 8. まとめ

- WatchKarte は「症状 → ベクトル検索 → LLM 診断」という RAG パイプラインで動いている
- 主要コストは **Cloudflare Workers（$5/月）＋ Groq API（従量課金）**
- D1・Vectorize・Workers AI はデータ件数・診断回数が少ないうちはほぼ無料レベル
- 10 万診断/月程度までは月 $15 未満、100 万診断/月でも $100 未満で運用可能な規模
- クレジットカード未登録でも、Cloudflare Workers Free プラン + Groq Free tier の範囲内であれば請求なく動作する
- ただし実運用では、D1・Workers の日次制限を避けるため Workers Paid プラン（$5/月）への加入が現実的

コストをさらに正確に見積もりたい場合は、実際に運用しながら Cloudflare ダッシュボードの使用状況と Groq の利用量を確認することをおすすめします。
